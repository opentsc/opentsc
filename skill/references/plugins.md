# OpenTSC Modules (formerly Plugins)

Modules are operational capabilities implemented as Agent Skills (SKILL.md). They are replaceable, independently testable, and scoped by the kernel bus (K6).

## Module contract

Each module is an Agent Skill with SKILL.md frontmatter declaring its contract:

```yaml
---
name: advisor.match
description: 任务-人选匹配建议引擎
vsm: S3
profession: 指挥官
reads: [entities, events, knowledge, judgment]
writes: [feedback]
binding: { kind: tool, ref: advisor_match }
model_tier: high
trigger: { kind: invoke }
sunset: { unused_after_months: 6 }
genesis: { template: advisor, domain: generic }
---
```

## Module families (interface slots)

### collector.* (S4 Perception)
Turns raw material into draft or confirmed intelligence events.
- `collector.manual` — user-dictated intelligence with Admiralty rating
- `collector.import` — batch import from files/exports
- `collector.contacts` — vCard/CSV import with dedup
- `collector.transcript` — meeting transcript ingestion
- `collector.platform.*` — X, Slack, Mattermost, email, calendar (stubs)
- `collector.hardware.*` — recording, IoT, wearable (stubs, edge-sanitized)

### scoring.* (S3)
Calculates scores reading all dimensions from judgment_codex (never hardcoded).
- `scoring.trust` — trust dimensions with review dates and expiry
- `scoring.suggestion` — quality scoring for reasoning chains

### knowledge.* (S3)
Three-layer knowledge with trigger conditions and auto-surfacing.
- `knowledge.facts` / `knowledge.methods` / `knowledge.principles`
- Granules must be auto-hit by advisors, not passively re-read from raw
- Unused granules sunset after N months

### advisor.* (S3 — system's judgment output)
Produces decisions and recommendations, always staged through K4.
- `advisor.match` — task-to-person matching using K7 attributes
- `advisor.relation` — relationship situation summary and action matrix
- `advisor.capability` — aggregates skills + mobilizable network
- `advisor.intake` — evaluates whether to accept external work
- `advisor.leverage` — value × accessibility × relationship_health − cost

### calibration.* (S4 — anti-echo-chamber)
- `calibration.review` — prompts for overdue prediction outcomes (session.idle)
- `calibration.stats` — hit rate by context/knowledge source/dimension
- `calibration.feedback` — outcome-proven patterns → candidate rules

### operations.* (S1)
Operations as entities with progress event streams.
- `operations.plan` / `operations.track` / `operations.review`
- Closing triggers mandatory review + calibration update

### briefing.* (S3 — read-only startup)
- `briefing.startup` — on session start: deadlines, due predictions, opportunities, conflicts, **and at least one uncomfortable item when present**

### recruiter.* (S4)
Detects capability gaps, searches for candidates, produces recruitment intents.

## Extension rules

- One module should do one narrow job.
- Declare reads/writes before implementation.
- Add sunset metadata for unused/superseded modules.
- New fields → register through K5 first.
- Cannot satisfy evidence requirements → output drafts, not facts.
- Modules generated by genesis engine must pass K8 validation gate + player approval.
